# Q1. Claude Code 如何处理"一次多个 tools"?

> 配套阅读:[../phase2-tool-system.md](../phase2-tool-system.md) §4 §5 · [索引](./question.md)
>
> **提问背景**:主笔记 §5 提了"流水线 vs 流式"两套执行模型,但没集中讲——当模型一次返回多个 `tool_use` 的时候,到底是怎么调度、并发、串行、收集结果的?

## 回答

一句话:**模型自主决定一轮发几个 `tool_use`,Claude Code 按工具自报的 `isConcurrencySafe(input)` 把它们切成连续批,read-only 段并发跑、非 read-only 段独占跑,顺序不重排,结果统一以多个 `tool_result` 块的形式回给模型。**

### 1. 多工具的源头:Anthropic API 的 tool_use 协议

模型在**一次 assistant message** 里就可以发 N 个 `tool_use` block:

```jsonc
{
  "role": "assistant",
  "content": [
    { "type": "text", "text": "Let me look at three files in parallel." },
    { "type": "tool_use", "id": "toolu_01", "name": "Read", "input": { "file_path": "a.ts" } },
    { "type": "tool_use", "id": "toolu_02", "name": "Read", "input": { "file_path": "b.ts" } },
    { "type": "tool_use", "id": "toolu_03", "name": "Grep", "input": { "pattern": "foo" } }
  ]
}
```

这是**模型自己的判断**——它觉得这几件事能同时做,就一次发出来。Claude Code 在 `query()` 解析后,把 `tool_use` blocks 数组丢给 orchestrator。

### 2. 默认路径:`runTools` + `partitionToolCalls`(batch 模型)

**入口**:`src/services/tools/toolOrchestration.ts:19-82`

```typescript
export async function* runTools(toolUseMessages, assistantMessages, canUseTool, toolUseContext) {
  let currentContext = toolUseContext
  for (const { isConcurrencySafe, blocks } of partitionToolCalls(toolUseMessages, currentContext)) {
    if (isConcurrencySafe) {
      // 并发跑这一批
      for await (const update of runToolsConcurrently(blocks, ...)) { yield ... }
      // contextModifier 在并发批里收集但不立即应用,等批结束统一应用
    } else {
      // 独占串行跑这一批(实际就只有 1 个 block)
      for await (const update of runToolsSerially(blocks, ...)) { yield ... }
    }
  }
}
```

**关键:`partitionToolCalls` 的切段规则**(`toolOrchestration.ts:91-116`):

```typescript
function partitionToolCalls(toolUseMessages, toolUseContext): Batch[] {
  return toolUseMessages.reduce((acc, toolUse) => {
    const tool = findToolByName(toolUseContext.options.tools, toolUse.name)
    const parsedInput = tool?.inputSchema.safeParse(toolUse.input)
    const isConcurrencySafe = parsedInput?.success
      ? Boolean(tool?.isConcurrencySafe(parsedInput.data))
      : false  // 解析失败时保守:不安全

    if (isConcurrencySafe && acc[acc.length - 1]?.isConcurrencySafe) {
      acc[acc.length - 1].blocks.push(toolUse)  // 加入上一批
    } else {
      acc.push({ isConcurrencySafe, blocks: [toolUse] })  // 新开一批
    }
    return acc
  }, [])
}
```

**重点 1:不重排,只切段**——如果模型发的是 `[Read, Read, Bash, Read]`,会切成:

```
批 1: [Read, Read]  (concurrency-safe → 并发)
批 2: [Bash]        (unsafe → 独占串行)
批 3: [Read]        (一个 Read 也是一批,但还是并发模式)
```

**为什么不全局重排把 3 个 Read 凑一起?** 因为模型可能依赖 `Bash` 的副作用——它可能写了文件、改了状态,后面那个 `Read` 是要读改动结果。**顺序就是模型的意图**,Claude Code 不擅自调换。

**重点 2:`isConcurrencySafe(input)` 是 per-input 的**,同一个工具不同入参可能不同结论:

| 工具 | 入参 | 是否并发安全 |
|---|---|---|
| `Read({file_path: 'a.ts'})` | ✓ | 只读,纯函数 |
| `Grep({pattern: 'foo'})` | ✓ | 只读 |
| `Bash({command: 'ls'})` | ✓(实际多数情况) | 视具体命令而定 |
| `Bash({command: 'rm a'})` | ✗ | 有副作用 |
| `Edit({file_path: 'a.ts'})` | ✗ | 写盘 |
| `Write({file_path: 'a.ts'})` | ✗ | 写盘 |

具体哪些工具/入参声明 safe,等 qa02 展开。

### 3. 并发批怎么跑

`runToolsConcurrently`(`toolOrchestration.ts:152-177`):

```typescript
async function* runToolsConcurrently(toolUseMessages, ...) {
  yield* all(
    toolUseMessages.map(async function* (toolUse) {
      toolUseContext.setInProgressToolUseIDs(prev => new Set(prev).add(toolUse.id))
      yield* runToolUse(toolUse, /* 匹配 assistantMessage */, canUseTool, toolUseContext)
      markToolUseAsComplete(toolUseContext, toolUse.id)
    }),
    getMaxToolUseConcurrency(),  // 默认 10,env CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY 可调
  )
}
```

**`all(generators, n)`** 是个并发池工具:同时拉 N 个 generator,谁先 yield 谁先出,**不保证模型发出顺序**——但这没关系,因为每个 `tool_result` 都带 `tool_use_id`,模型靠 id 配对,顺序无所谓。

### 4. 串行批怎么跑

`runToolsSerially`(`toolOrchestration.ts:118-150`):

```typescript
async function* runToolsSerially(toolUseMessages, ...) {
  let currentContext = toolUseContext
  for (const toolUse of toolUseMessages) {
    for await (const update of runToolUse(toolUse, ..., currentContext)) {
      if (update.contextModifier) {
        currentContext = update.contextModifier.modifyContext(currentContext)
        // ↑ 关键:前一个工具改的 ctx,后一个工具看得见
      }
      yield { message: update.message, newContext: currentContext }
    }
  }
}
```

**`contextModifier` 的链式应用是串行批的灵魂**——`EnterPlanMode` / `EnterWorktree` 这类"改运行环境"的工具靠它工作:它们的 `ToolResult.contextModifier` 把后续 ctx 改成新模式,从下一个工具开始生效。

**`Tool.ts:329` 明确注释**:

```typescript
// contextModifier is only honored for tools that aren't concurrency safe.
contextModifier?: (context: ToolUseContext) => ToolUseContext
```

——并发批里就算工具返回了 `contextModifier`,也**不会**被应用。理由:多个工具同时改 ctx,顺序不定 + 结果不确定,直接放弃。

### 5. 替代路径:`StreamingToolExecutor`(边 stream 边跑)

`src/services/tools/StreamingToolExecutor.ts:40-150` —— 模型还在生成 `tool_use` block 的时候就开跑:

```typescript
class StreamingToolExecutor {
  private tools: TrackedTool[] = []
  private siblingAbortController: AbortController  // 同批 Bash 错杀

  addTool(block, assistantMessage) {
    const tool = findToolByName(this.toolDefinitions, block.name)
    const parsed = tool?.inputSchema.safeParse(block.input)
    const isConcurrencySafe = parsed?.success ? tool.isConcurrencySafe(parsed.data) : false
    this.tools.push({ id, block, status: 'queued', isConcurrencySafe, pendingProgress: [] })
    void this.processQueue()
  }

  canExecuteTool(isConcurrencySafe) {
    const executing = this.tools.filter(t => t.status === 'executing')
    return executing.length === 0
        || (isConcurrencySafe && executing.every(t => t.isConcurrencySafe))
  }
}
```

跟 batch 模型相比:

| 维度 | `runTools`(batch) | `StreamingToolExecutor` |
|---|---|---|
| 触发时机 | 模型 stream 完才开跑 | block 一来就入队 |
| 并发控制 | 一次性 partition 后跑 | 在线状态机,逐块判定 |
| 顺序 | 保模型发出顺序切批,批内并发 | 同样按模型发出顺序入队,canExecuteTool 决定何时起 |
| abort 隔离 | 共享 parent abort | **`siblingAbortController` 独立**——Bash 错只杀同批 Bash,不让 parent turn 结束 |
| 失败回退 | 重新切 batch | `discard()` 整批丢 |
| 收益 | 实现简单 | **延迟更低**(LLM 还在 stream 时 grep 已经跑完了) |

二者的契约一致:都靠 `isConcurrencySafe(input)` 切并发,都最终走 `runToolUse → tool.call`。

### 6. 结果怎么回给模型

每个 `runToolUse` yield 出 **一条 `UserMessage`**,内容是一个 `tool_result` block:

```typescript
createUserMessage({
  content: [{ type: 'tool_result', tool_use_id: toolUse.id, content: '...' }],
  toolUseResult: '...',
  sourceToolAssistantUUID: assistantMessage.uuid,
})
```

到了发回给 API 时,`normalizeMessagesForAPI` 把**同一轮内的多条 tool_result UserMessage 合并成一条 user message,多个 content blocks**:

```jsonc
{
  "role": "user",
  "content": [
    { "type": "tool_result", "tool_use_id": "toolu_01", "content": "..." },
    { "type": "tool_result", "tool_use_id": "toolu_02", "content": "..." },
    { "type": "tool_result", "tool_use_id": "toolu_03", "content": "..." }
  ]
}
```

API 要求:**所有 `tool_use_id` 必须有对应的 `tool_result`**(否则模型卡死),顺序无关——靠 id 配对。

### 7. 边界情况

| 场景 | 处理 |
|---|---|
| 模型发了不存在的工具名 | `runToolUse` 早返 `"No such tool available: X"` 包成 `tool_use_error`;**其它工具照常跑** |
| 某个工具 input schema 不过 | Zod 拒,包成 `InputValidationError`;**仅这一个失败**,不影响同批其它 |
| Bash 在并发批中失败 | streaming 路径下,`siblingAbortController.abort()` 杀同批 Bash;batch 路径下其它工具继续跑完 |
| 工具 `call()` 抛异常 | `runToolUse` 外层 `try/catch` 包成 `<tool_use_error>...</tool_use_error>`;**不冒泡到 query 主 loop**,turn 不死 |
| 模型一轮发 100 个 tool_use | 并发池 maxConcurrency=10,排队消化;Streaming 按模型 stream 节奏陆续入队 |
| 串行批中前一个工具的 `contextModifier` | 应用到 `currentContext`,后面工具看到新 ctx(典型:`EnterPlanMode` 之后所有工具受 plan-mode 影响) |
| 并发批中工具返回了 `contextModifier` | **被静默忽略**(Tool.ts:329 注释) |
| 用户在工具跑一半发了新消息 | 工具的 `interruptBehavior?(): 'cancel' \| 'block'` 决定——cancel 杀掉 + 丢结果,block 让新消息等 |
| 同一轮里 read 和 write 混合 | 严格按发出顺序切批:`[R, R, W, R, R]` → `[R,R]` ‖ `[W]` → `[R,R]` ‖,绝不把后面的 R 提前到 W 之前 |

## 对我们后端的启示

| Claude Code 做法 | 后端可借鉴 |
|---|---|
| 模型一轮发多 tool_use,**保留顺序切段** | 不要拆成"一次只 call 一个 tool 的 N 轮"——浪费 LLM round-trip。一次性收齐,按 safety 分批跑完再回 |
| `isConcurrencySafe(input)` per-input 自报 | 接口设计就允许同名工具不同入参不同结论。例:HTTP GET 默认 safe,但同一 URL 1 秒内 100 次 GET 可能要限,这种 detection 工具自己最清楚 |
| `Math.max(env, default=10)` 控并发上限 | 用 semaphore 限并发,默认值要可调(LLM 偶尔会爆 100 个 tool_use,需要兜底) |
| 并发批**忽略** `contextModifier` | 简单粗暴但正确:多工具同时改 ctx 必冲突。我们若有"修改 ctx 的工具",定为 not-concurrency-safe |
| streaming 路径用**子 abort controller** | 局部失败局部止血,主 abort 不动 turn 还能继续。`AbortController.signal` 链式结构在 Node 里就有,直接抄 |
| 工具失败包成 `tool_use_error` block 给模型 | 不要把工具异常冒到 agent loop 外——模型看到 `tool_result(is_error=true)` 后能自己决定重试/换路。这是"模型容错"的核心信道 |
| 顺序敏感的 `[R, R, W, R, R]` 不被全局重排 | 性能可能差(少了一些可并发的空间)但语义正确。我们若做"激进重排"需要数据依赖分析,得不偿失 |
| 多个 `tool_result` 合并到一条 user message | 是 API 协议要求,但实现上简化了"哪些 tool_result 属于本轮"的归属判断——同 turn 的就是同 user message |
