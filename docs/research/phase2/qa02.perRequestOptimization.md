# Q2. 请求模型之前,有"优化一次请求"的函数吗?是如何完成的?

> 配套阅读:[../phase2-tool-system.md](../phase2-tool-system.md) · [../phase1/qa07.preprocessingPipeline.md](../phase1/qa07.preprocessingPipeline.md) · [索引](./question.md)
>
> **提问背景**:[qa01](./qa01.multipleToolsHandling.md) 讲了"模型返回多个 `tool_use` 怎么调度",但模型为什么能返回多个工具调用?调它之前都做了什么?

## 回答

### 0. 先确认一件事:工具调用是模型自主产生的

是的。每一轮 LLM 响应里的 `tool_use` block **由模型自己生成**——它根据请求里给它的 `tools` 列表 + system_prompt + messages 自主决定调用哪些、传什么参数。Claude Code 只负责:

- 让模型**看到哪些工具**(填 `tools` 字段)
- 给它**当前上下文**(填 `messages` 字段)
- 给它**任务指令**(填 `system_prompt` 字段)
- **解析 + 执行**它回的 `tool_use`
- **不替模型决定调哪个工具**——除了通过 deny rule "把不该看见的工具藏起来"

所以"请求模型"是 agent loop 的**核心动作**,而"请求前的优化"决定了模型看到的世界有多干净、多省 token、多稳。

### 1. 单次请求的全景:`query()` 内部一次循环

入口函数 `src/query.ts`(同时是个 `async function*`)的 `while(true)` 循环里,每一圈对应**一次 LLM 调用**。每一圈的结构:

```
        ┌──────────────────────────────────────────┐
        │       一圈 = 一次 callModel(...)           │
        ├──────────────────────────────────────────┤
        │                                          │
        │ 【消息层优化】(5 层 + 闸门,见 qa07)         │
        │  ① applyToolResultBudget    工具结果落盘  │
        │  ② snip                     消息直接丢    │
        │  ③ microcompact             tool_result 清空 │
        │  ④ contextCollapse          归档投影    │
        │  ⑤ autocompact              整段 summary  │
        │  ⑥ blocking_limit gate      硬闸合成 PTL  │
        │                                          │
        │ 【工具层优化】                             │
        │  ⑦ assembleToolPool         内置作前缀 +  │
        │                              MCP 追加 +    │
        │                              uniqBy        │
        │  ⑧ Tool schema deferral     ToolSearch 过滤  │
        │                              未发现的 schema 不发  │
        │                                          │
        │ 【系统提示层优化】                          │
        │  ⑨ system_prompt 拼装       cache_control 锚点  │
        │  ⑩ user_context prepend     动态附件      │
        │  ⑪ beta headers 装配        Tool Search header │
        │                              + cache editing  │
        │                                          │
        │ 【模型选择 + 模式状态】                      │
        │  ⑫ getRuntimeMainLoopModel  超 200K 自动 │
        │                              切大窗口模型 │
        │  ⑬ fastMode / effortValue   AppState 注入 │
        │                                          │
        │ 【发出】                                    │
        │  ⑭ callModel({messages, systemPrompt,     │
        │              tools, signal, options})     │
        │      └→ queryModelWithStreaming           │
        │      └→ HTTP streaming                    │
        │                                          │
        │ 【回收 + 自愈】                             │
        │  ⑮ tombstone-on-streaming-fallback        │
        │  ⑯ withhold-then-recover (PTL/max_output) │
        │  ⑰ usage 累加 / cache 命中率上报           │
        │                                          │
        └──────────────────────────────────────────┘
                          │
                          ▼
                  下一圈 or 退出
```

### 2. 一步步看

#### 2.1 消息层(5 层 + 闸门)

详见 [Phase 1 qa07](../phase1/qa07.preprocessingPipeline.md)。一句话:**代价从低到高串联,前面够用就别动后面**——`query.ts:365-468` 这 60 行就是流水线全部,每段有 "runs BEFORE / runs AFTER" 注释。

跟"优化请求"直接相关的两点:
- ① `applyToolResultBudget`(`query.ts:379`):单工具结果超 `maxResultSizeChars` 落盘 + 替换为文件引用——**最便宜的省 token 手段**,工具级别就解决
- ⑥ `blocking_limit`(`query.ts:637-648`):硬闸,超了直接 yield 合成 `PROMPT_TOO_LONG_ERROR_MESSAGE` **不发 API**——给反应式压缩留头部空间

#### 2.2 工具层 ⭐(本 qa 的重头)

##### ⑦ `assembleToolPool` — cache-friendly 排序

`src/tools.ts:345-367`:

```typescript
return uniqBy(
  [...builtInTools].sort(byName).concat(allowedMcpTools.sort(byName)),
  'name',
)
```

注释揭示一个真实约定:

> The server's `claude_code_system_cache_policy` places a global cache breakpoint after the last prefix-matched built-in tool; a flat sort would interleave MCP tools into built-ins and invalidate all downstream cache keys.

→ **内置工具排在前面当连续前缀,MCP 工具排在后面**。这样新接一个 MCP server 不会冲掉内置段 cache,只刷新 MCP 段——**这是单请求最重要的 cache 友好性优化**。

##### ⑧ Tool schema 按需暴露(ToolSearch)

`src/services/api/claude.ts:1120-1172`:

```typescript
let useToolSearch = await isToolSearchEnabled(model, tools, getToolPermCtx, agents, 'query')

const deferredToolNames = new Set<string>()
if (useToolSearch) {
  for (const t of tools) {
    if (isDeferredTool(t)) deferredToolNames.add(t.name)
  }
}

// 没有 deferred 工具且 MCP 全连完 → 关掉 ToolSearch(避免空转)
if (useToolSearch && deferredToolNames.size === 0 && !hasPendingMcpServers) {
  useToolSearch = false
}

if (useToolSearch) {
  const discoveredToolNames = extractDiscoveredToolNames(messages)
  filteredTools = tools.filter(tool => {
    if (!deferredToolNames.has(tool.name)) return true        // 非 deferred 全留
    if (toolMatchesName(tool, TOOL_SEARCH_TOOL_NAME)) return true  // ToolSearch 自己永留
    return discoveredToolNames.has(tool.name)                 // deferred 只留已发现的
  })
} else {
  filteredTools = tools.filter(t => !toolMatchesName(t, TOOL_SEARCH_TOOL_NAME))
}
```

**核心机制**:`extractDiscoveredToolNames(messages)` 扫一遍 message history,所有"模型曾经 `ToolSearch({query: 'select:Foo'})` 成功过"的 Foo 都进 discovered 集合。**只有 discovered 集合里的 deferred 工具,这次才发 schema**——`tools` 字段瘦下来。

工具数量 ~100+ 时省的就是几万 tokens 的 schema + 模型注意力。

##### 工具 schema 装配本身

`claude.ts:1235` 起的 `toolSchemas = await Promise.all(filteredTools.map(toolToAPISchema(...)))` 把每个 `Tool` 转成 API 要的 JSON,其中 deferred 工具在 schema 里加 `defer_loading: true` 字段——告诉 API 只发名字描述,不发参数 schema。

#### 2.3 系统提示层

##### ⑨ `fullSystemPrompt` 拼装(`query.ts:449-451`)

```typescript
const fullSystemPrompt = asSystemPrompt(
  appendSystemContext(systemPrompt, systemContext),
)
```

`systemPrompt` 是**静态 + cache-friendly 的指令文本**(包含 `memoryMechanicsPrompt` 这种"使用说明书",见 [qa04](../phase1/qa04.memoryMechanicsPrompt.md));`systemContext` 是动态会话级补充。两者拼起来还要在尾部插 `cache_control` 标记——告诉 API 这部分可以缓存。

##### ⑩ user_context 前置(`query.ts:660`)

```typescript
messages: prependUserContext(messagesForQuery, userContext),
```

`userContext` 包括:
- 自动 memory 内容(`MEMORY.md` 和 `CLAUDE.md` 读出来的)
- 附件(`@file` 展开的)
- 任务级元数据

**前置而非塞 system_prompt**——理由见 [qa04](../phase1/qa04.memoryMechanicsPrompt.md):memory 内容是动态的,塞 system_prompt 会破 cache,前置到 user message 反而不破。

##### ⑪ beta headers 装配

`claude.ts:1174-1182`:

```typescript
const toolSearchHeader = useToolSearch ? getToolSearchBetaHeader() : null
if (toolSearchHeader && getAPIProvider() !== 'bedrock') {
  if (!betas.includes(toolSearchHeader)) betas.push(toolSearchHeader)
}
```

`defer_loading: true` 这种字段需要 beta header 才被 API 接受。**1P/Foundry 用 `advanced-tool-use`,Vertex/Bedrock 用 `tool-search-tool`,Bedrock 还得放 `extraBodyParams` 而非 `betas`**——provider 差异在这一层 normalize。

类似的还有 cache editing (`CACHE_EDITING_BETA_HEADER`) 等。

#### 2.4 模型选择 + 模式

##### ⑫ `getRuntimeMainLoopModel`(`query.ts:572-578`)

```typescript
let currentModel = getRuntimeMainLoopModel({
  permissionMode,
  mainLoopModel: toolUseContext.options.mainLoopModel,
  exceeds200kTokens:
    permissionMode === 'plan' &&
    doesMostRecentAssistantMessageExceed200k(messagesForQuery),
})
```

**plan 模式 + 最近 assistant 超 200K → 自动切到大窗口模型**。这是 per-request 的运行时模型选择——同一会话不同请求可能用不同模型。

##### ⑬ 其他注入字段

```typescript
options: {
  model: currentModel,
  ...(config.gates.fastModeEnabled && { fastMode: appState.fastMode }),
  effortValue: appState.effortValue,
  advisorModel: appState.advisorModel,
  skipCacheWrite,
  taskBudget: { total, remaining },
  hasPendingMcpServers,
  ...
}
```

**`hasPendingMcpServers`** 这个字段尤其重要:**让 API 知道 MCP 还在连,即使现在没有 deferred 工具,ToolSearch 也别关掉**——等 MCP 连上后模型可以现搜。

#### 2.5 发出:`deps.callModel` → `queryModelWithStreaming`

`query.ts:659-708`:

```typescript
for await (const message of deps.callModel({
  messages: prependUserContext(messagesForQuery, userContext),
  systemPrompt: fullSystemPrompt,
  thinkingConfig: toolUseContext.options.thinkingConfig,
  tools: toolUseContext.options.tools,  // ← 这里 tools 还是全集!
  signal: toolUseContext.abortController.signal,
  options: { ... },
})) {
  ...
}
```

注意:**这里传给 callModel 的 `tools` 是工具全集**,真正的 `filteredTools`(经过 ToolSearch 过滤的)是 `callModel` 内部(`claude.ts:1120`)再做的——目的是让 `callModel` 同时能拿到全集(给 ToolSearch 的 prompt 列举所有可用 MCP 工具)和过滤后的(给 API 用)。

`deps.callModel` 默认指向 `queryModelWithStreaming`(`src/query/deps.ts:23, 36`),实现在 `src/services/api/claude.ts`——HTTP streaming + 重试 + usage 统计的真正发出点。

#### 2.6 回收 + 自愈

##### ⑮ Streaming fallback tombstone(`query.ts:712-740`)

streaming 中途失败 → 触发非 streaming 回退。这时已经 yield 出去的 partial assistant message(尤其 thinking blocks)签名不合法,**回放会 API 报错**。解决:yield `{ type: 'tombstone', message }` 让 UI / transcript 把这条消息抹掉,然后清空 `assistantMessages` / `toolResults` / 重建 `StreamingToolExecutor`,重新跑。

##### ⑯ Withhold-then-recover(`query.ts:1175-1246`)

API 返回 `prompt_too_long` 或 `max_output_tokens` 时:

- **withhold**:streaming 阶段先**不**把这条错误消息往 SDK callers 发——因为这条消息含有部分 partial 输出
- **recover**:走恢复流程(`prompt_too_long` 调 autocompact/snip,`max_output_tokens` 注入"please continue"提示让模型续写)
- **agree**:withhold 和 recover 两个子系统的开关必须一致(`query.ts:622-625` 注释明确)——CACHED_MAY_BE_STALE 会在 5-30s 的 stream 中漂移,withhold 但不 recover 会"吃掉"这条消息

这是 **per-request 级别**的失败自愈,**比 turn 级或 session 级 retry 早得多**。

##### ⑰ usage 累加 + cache 命中率

每次成功 callModel 后:
- `usage.input_tokens` / `output_tokens` 累加到 `cumulativeUsage`
- `cache_read_input_tokens` / `cache_creation_input_tokens` 单独跟踪
- `notifyCompaction(querySource)` 提示 cache-break 检测器"刚刚做了压缩,别报警"——见 [qa07](../phase1/qa07.preprocessingPipeline.md) §5 mutual exclusion

### 3. "一次请求"的成本控制总结表

| 层 | 优化点 | 何时省 | 代价 |
|---|---|---|---|
| 工具结果落盘 | `maxResultSizeChars` per-tool | 单工具结果太大 | 几 ms 写盘 |
| snip | 直接丢老消息 | 主动 / 时间 gap | 几乎为 0 |
| microcompact | tool_result 清空 / cache 编辑 | tool_result 太多 | 几乎为 0 / 一次 cache_edits |
| contextCollapse | 渐进归档投影 | 长会话累积 | 一些 commit 开销 |
| autocompact | 整段 summary | 阈值前 | 一次 forked LLM 调用(贵) |
| **assembleToolPool 排序** | **保 cache breakpoint** | **每次请求** | **零** |
| **ToolSearch 过滤** | **只发已发现 deferred 工具的 schema** | **工具池大时** | **零(消息持久化)** |
| system_prompt cache 标记 | cache_control 锚点 | 每次请求 | 零 |
| user_context 前置 | 避开 system_prompt cache | 每次请求 | 零 |
| 模型切换 | 超 200K 切大窗口 | 大文档场景 | 模型成本上升 |
| withhold-then-recover | 不暴露 partial 错误 | 流式失败 | 一次重试 |
| streaming fallback | 切非流式 | streaming 协议错 | 一次重试 + tombstone |

## 对我们后端的启示

| Claude Code 做法 | 后端可借鉴 |
|---|---|
| **"优化请求"不是单一函数,是 turn loop 里每圈的一整套 stages** | 不要把"省 token"和"发请求"分开管;在发请求前用 stage pipeline 串联(成本递增) |
| `assembleToolPool` 保**内置作连续前缀** | 工具列表的稳定性 = cache 命中率。新接入的扩展工具应**追加在尾部**,绝不要插中间 |
| ToolSearch 把 schema 状态做成**消息持久化** | 不需要任何外部存储——message history 本身就是"已发现工具集"。断点续传天然支持 |
| `hasPendingMcpServers` 字段让 ToolSearch 等连接完成 | 异步资源加载时,**不要急于关掉发现机制**——状态字段而非时间窗口 |
| `getRuntimeMainLoopModel` per-request 切模型 | 单会话可以不同请求用不同模型(长文档切大窗口、简单问答切小模型) |
| **withhold-then-recover**:partial 错误不外露 | 错误自愈层级:per-request → per-turn → per-session,**优先 per-request**——别把流式中的瞬态错误冒到外层 |
| `tombstone` 消息让 UI 干净回退 | streaming 失败重置时,**显式告诉下游"刚才那条不算"**,而不是依赖下游自己判 |
| `prependUserContext` 而非塞 system_prompt | 动态内容前置 user,保护 system 的 cache prefix |
| beta header / extraBodyParams 的 **provider 差异化 normalize** | 抽象一层"API 协议适配",别让上层关心 1P/Vertex/Bedrock 差异 |
| `notifyCompaction` 主动告知 cache 检测器 | 自己造成的 cache miss **要主动通知监控**,避免误报 |

### 一句话核心

Claude Code 的"优化一次请求"不是某个 `optimizeRequest()` 函数,而是 `query()` 每圈的 ~10 个**有序阶段**:**先压消息、再装工具池、再过滤 schema、再拼 prompt、再选模型、再发出、再自愈**。每个阶段单独便宜、组合起来既省 token 又保 cache;阶段顺序由 cost 和依赖共同决定,任何一处错位都对应一个线上事故注释。
