# Q3. 那 `system_init` 是不是就是 mcp / tools / skills 这些"工具组"?

> 配套阅读:[../phase1-agent-loop.md](../phase1-agent-loop.md) · [qa02.systemInit](./qa02.systemInit.md) · [索引](./question.md)
>
> **提问背景**:[Q2](./qa02.systemInit.md) 之后的追问,想压缩成"能力清单"这一个概念。

## 回答

**核心理解对,但稍微窄了**。`system_init` 里的字段可以分成**三大类**,工具组只是其中最显眼的一类:

### 三大类字段

```
┌─ ① 能力清单(用户问的"工具组") ──────────────────────┐
│  tools          — 内置工具名列表(Bash / FileRead / …) │
│  mcp_servers    — 已连接的 MCP server 及状态           │
│  slash_commands — 用户可调用的 / 命令名                │
│  agents         — 可用的子 agent type                  │
│  skills         — 已加载的 skill 名                    │
│  plugins        — 已启用的 plugin(name/path/source)   │
└────────────────────────────────────────────────────────┘
┌─ ② 运行环境 ──────────────────────────────────────────┐
│  cwd            — 当前工作目录                         │
│  session_id     — 会话 ID                              │
│  uuid           — 这条 init 消息自身的 UUID            │
│  claude_code_version — CLI 版本                        │
│  apiKeySource   — 'env' / 'config' / 'keychain' …      │
└────────────────────────────────────────────────────────┘
┌─ ③ 会话策略/模式 ────────────────────────────────────┐
│  model          — 本轮使用的模型                       │
│  permissionMode — default / plan / acceptEdits / bypass│
│  output_style   — 输出风格(default / explanatory / …) │
│  betas          — 服务端 beta 标签列表                 │
│  fast_mode_state — Fast 模式开关状态                   │
└────────────────────────────────────────────────────────┘
```

(`messaging_socket_path` 还有一个 ant-only 的 UDS 路径,Anthropic 内部用,外部构建编译掉)

### 为什么这三类要一起放进 init

它们都满足同一个性质:**"客户端 UI 需要,但不该从 LLM 输出里 parse"**。

| 字段类 | 客户端拿来干嘛 |
|---|---|
| **能力清单** | 渲染选择器(`@agent-xxx` 下拉、`/` 命令补全、工具开关页面) |
| **运行环境** | 标题栏显示 cwd、session 链接、版本号告警(升级提示) |
| **会话策略** | 模型徽标 / "Plan 模式"横幅 / 风险操作弹窗的提示语 / 根据 betas 决定要不要露出某个新功能开关 |

`buildSystemInitMessage` 的注释原话(`systemInit.ts:42-44`):

> session metadata (cwd, tools, model, commands, etc.) that remote clients use to **render pickers and gate UI**

"render pickers" 对应能力清单,"gate UI" 同时覆盖了运行环境(版本太老就提示)和会话策略(plan 模式就禁用 Bash 按钮)。

### 一个更精确的概念

与其叫"工具组",不如叫:

> **`system_init` = "这次 turn 的能力清单 + 环境标识 + 策略快照",是客户端把会话状态摆出来给用户看的元数据底座。**

记住一个反例就够了:**`session_id` 和 `model` 也在里面**,这两个明显不是"工具"。

## 对照源码

- `src/utils/messages/systemInit.ts:53-96` — 完整字段列表
- `src/utils/messages/systemInit.ts:23-25` — `sdkCompatToolName`:工具改名的向后兼容映射(`Agent` → `Task`)
- `src/utils/messages/systemInit.ts:42-44` — "render pickers and gate UI" 注释

## 对我们后端的启示

(在 [Q2](./qa02.systemInit.md) 的基础上补一条)

| Claude Code 做法 | 后端可借鉴 |
|---|---|
| **把 init 拆成三类**(能力清单 / 环境 / 策略)统一一帧下发 | 我们的 SSE/WebSocket 第一帧除了工具列表,也别忘了带 `session_id` / `model` / `permission_mode` 这些**前端 UI 状态**。不然前端就只能"问一次发一次",来回成本高 |
| **`betas` 字段做客户端 feature gating** | 后端可以发一份 "服务端启用的能力标签" 给前端,前端按标签 gate UI——**比让前端自己嗅探服务版本**更可靠 |
