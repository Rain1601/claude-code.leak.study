# Q6. `QueryEngine.submitMessage()` 这块就是"构建提示词的流程",输出最终提示词后 call LLM 吗?

> 配套阅读:[../phase1-agent-loop.md](../phase1-agent-loop.md) 第 0 节流程图 · [qa02.systemInit](./qa02.systemInit.md) · [索引](./question.md)
>
> **提问背景**:Phase 1 笔记第 0 节流程图把 `submitMessage` 那块画成了准备 system_prompt / processUserInput / system_init 等等步骤,直觉上容易理解为"一次性构建好最终提示词 → 喂给 LLM"。

## 回答

**不对。三个误解需要拆开**:

1. ❌ "submitMessage 输出最终提示词" → ✅ **没有"最终提示词"这种单一对象**。`system_prompt` 和 `messages` 是两条独立的轴,API request body 也是把它们分两个字段发(`system: "..."`, `messages: [...]`),不存在中间合并那一步。
2. ❌ "submitMessage 接着 call LLM" → ✅ **submitMessage 自己不直接 call LLM**。它准备好参数后**委托给 `query()`**,LLM call 是 `query.ts` 里干的。
3. ❌ "一次 submitMessage = 一次 LLM call" → ✅ **一次 submitMessage = 一个 turn = N 次 LLM call**。模型每说一次话,可能要工具 → 执行 → 再问模型 → 又要工具 → … 直到模型说"我说完了"。**这个 N 在 submitMessage 调用时未知,由模型一边跑一边决定**。

## 正确的心智模型

把 `submitMessage` 和 `query` 的关系画清楚就懂了:

```
QueryEngine.submitMessage(prompt)   ← 异步生成器,一个 turn 的"外壳/启动器"
│
├─ ① 一次性准备 turn 入口参数
│    • 拼 system_prompt(default + memory + appendSystemPrompt)
│    • processUserInput → 处理 slash/附件/模型覆盖,产出 messages
│    • 拿到当前 cwd、permission_mode、可用 tools/MCP/agents/skills
│
├─ ② 一次性 yield 一条 system_init 给 SDK 客户端(UI 渲染用)
│
├─ ③ 委托 → query(prompt 准备好的参数)            ← 真正的 turn 引擎
│   ┌─────────────────────────────────────────────────────────┐
│   │ while (true) {                                          │
│   │   messagesForQuery = preprocess(messages)               │
│   │      ↑ snip / microcompact / collapse / autocompact     │
│   │      ↑ 每次迭代都重新跑一遍                              │
│   │                                                         │
│   │   for await (msg of deps.callModel({                    │
│   │       systemPrompt,         ← 这个轴几乎不变              │
│   │       messages: messagesForQuery,  ← 这个轴每轮都变      │
│   │       tools, …                                          │
│   │   })) { … }                  ← ⭐ 真正的 LLM call,每 turn 跑 N 次 │
│   │                                                         │
│   │   if (!needsFollowUp) return Terminal                   │
│   │   runTools(toolUseBlocks) → 拿到 tool_results          │
│   │   messages.push(...assistantMessages, ...toolResults)  │
│   │   continue                                              │
│   │ }                                                       │
│   └─────────────────────────────────────────────────────────┘
│
├─ ④ 边消费 query 的 yield,边做"账本"工作:
│    • 累加 usage(input/output/cache tokens)
│    • turnCount++、检查 maxTurns
│    • 检查 maxBudgetUsd($ 消耗)
│    • 检查结构化输出重试次数
│    • 包装 canUseTool 收集 permission denial
│    • 把内部 Message 翻译成 SDKMessage yield 出去
│    • 持久化 transcript 到磁盘
│
└─ ⑤ 终态合成 result 消息
     • { subtype: 'success', usage, cost, num_turns, … }
     • 或 'error_max_turns' / 'error_max_budget_usd' / …
```

### `submitMessage` 自己做什么 vs 不做什么

| 维度 | submitMessage 做 | submitMessage **不**做 |
|---|---|---|
| **system_prompt** | 一次性拼好,传给 query | 不发请求 |
| **messages 预处理** | 用 processUserInput 把这次用户输入加工成 messages 列表 | 不做压缩/snip(那些每轮 turn iteration 干一次,在 `query.ts:365-543`) |
| **system_init** | yield 一条给客户端 UI | 不发给模型 |
| **LLM 调用** | ❌ | 在 `query.ts:659` 的 `deps.callModel({...})` |
| **turn 内的 N 轮迭代** | ❌ | `query.ts:307` 的 `while (true)` |
| **工具执行** | ❌ | `query.ts:1382` 的 `runTools(...)` |
| **错误恢复**(prompt_too_long、fallback model) | ❌ | `query.ts:1066-1256` |
| **会话记账**(usage、cost、denial、turnCount) | ✅ | — |
| **持久化 transcript** | ✅ | — |
| **terminal → SDK result 消息** | ✅ | — |

### "提示词"不是对象,是**每次 callModel 时基于两条轴动态拼的请求体**

每次 `query.ts:659` 调 `deps.callModel({...})`,API request body 大致是:

```
{
  "model": "claude-opus-4-7",
  "system": <systemPrompt>,        ← 这个 turn 里几乎不变(可能因为压缩重新组合一点)
  "messages": <messagesForQuery>,  ← 每次都不同!每轮迭代重新走过预处理
  "tools": [...],
  "tool_choice": …,
  …
}
```

`messagesForQuery` 在 `query.ts:365` 每次迭代都**重新构造**:
```ts
let messagesForQuery = [...getMessagesAfterCompactBoundary(messages)]
// ↓ 然后依次走过:
//   • applyToolResultBudget(超量工具结果转文件引用)
//   • snip(移除僵尸消息)
//   • microcompact(局部缓存编辑)
//   • contextCollapse(段落折叠)
//   • autocompact(整体摘要)
```

所以同一个 turn 里,**第 1 次和第 3 次 callModel 看到的 messages 都不一样**(中间插了 tool_results,可能触发了某层压缩)。**根本没有一份"最终提示词"在那里**——只有每次调用临时拼出来的 request body。

## 更精确的概括

| 你之前的描述 | 修正后 |
|---|---|
| "构建提示词的流程,输出最终提示词,然后 call LLM" | "**Turn 启动器**:一次性准备好 turn 入口参数(system_prompt 拼装 + user input 加工 + 元数据握手),然后**委托给 `query()` 跑 turn 内的 N 轮 LLM 调用循环**,自己只在外面做账本和翻译" |

更短一句:

> `submitMessage()` 是 turn 的**总线**,不是 LLM 的**调用器**。

## 对我们后端的启示

| Claude Code 做法 | 后端可借鉴 |
|---|---|
| **"会话外壳"和"turn 引擎"分两层**(`QueryEngine` vs `query()`) | 后端应当**显式区分**:"会话级"的事(usage 累加、budget 检查、持久化、对外协议翻译)留在外层;"turn 内"的事(LLM 调用循环、工具调度、压缩、错误恢复)封进内层。**两层都用异步生成器,通过 yield 协作而非函数调用栈** |
| **system_prompt 在 turn 入口拼一次,turn 内反复用** | 我们的 API 里,**system prompt 应当在请求入口立即固化**,后续任何一轮 LLM call 都用同一份,**不要在 turn 内重算**——既贵又会让 prompt cache 失效 |
| **messages 在每次 LLM call 前重新预处理** | 我们的 turn loop 在调 LLM 前**总是**走一遍"消息流水线"(去除/压缩/注入附件),而**不是入口处理一次后整 turn 复用**——上下文是个动态量,新 tool_result 注入后可能就触发压缩了 |
| **API request body 始终是"system + messages + tools"三条轴,不预合并** | 不要发明"最终提示词"这种合并对象。**LLM API 的协议本身就分轴**(system 字段和 messages 字段),你的内部抽象也对齐它,以后切 provider、加 cache 标记都更容易 |
| **一次 turn = N 次 LLM call,N 由模型决定** | 任何对"一次请求 = 一次 LLM call"的设计假设都要**主动打破**。后端 API 的成本/超时/限流模型都该按 turn(可能内含多次 call)算,而不是按单次 call 算 |
