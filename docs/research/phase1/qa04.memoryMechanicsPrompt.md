# Q4. `system_prompt` 里的 `memoryMechanicsPrompt` 是什么?为什么记忆里面会有 prompt?

> 配套阅读:[../phase1-agent-loop.md](../phase1-agent-loop.md) · [qa02.systemInit](./qa02.systemInit.md) · [索引](./question.md)
>
> **提问背景**:[Q2](./qa02.systemInit.md) 里贴的 `system_prompt` 拼装代码片段:
>
> ```ts
> const systemPrompt = asSystemPrompt([
>   ...(customPrompt !== undefined ? [customPrompt] : defaultSystemPrompt),
>   ...(memoryMechanicsPrompt ? [memoryMechanicsPrompt] : []),
>   ...(appendSystemPrompt ? [appendSystemPrompt] : []),
> ])
> ```
>
> "memory" + "prompt" 凑在一起容易让人误解成"记忆里存的 prompt"。

## 回答

### 名字误导,本质是"记忆系统的使用说明书"

`memoryMechanicsPrompt` 是教模型**怎么使用记忆系统**的指令文本——**记忆的"机制说明",不是记忆的内容**。

- **机制说明**(memory mechanics)= "记忆系统在哪、有哪些类型、什么时候存、什么时候不存、用什么 frontmatter 格式、`MEMORY.md` 怎么当索引……" → **进 system_prompt**
- **记忆内容**(用户实际记的东西,比如 `user_role.md` 写了"用户是 Go 工程师")→ **不进 system_prompt**,通过 attachment(`nested_memory` / `relevant_memories`)在 turn 里**按需注入 user context**

源码里那行注释说得很直接(`src/memdir/memdir.ts:196-197`):

> Used by both `buildMemoryPrompt` (agent memory, includes content) and **`loadMemoryPrompt` (system prompt, content injected via user context instead).**

`loadMemoryPrompt`(走进 system_prompt 这条路径)**只放说明书,不放任何记忆条目本身**。

### 这份"说明书"长什么样

`buildMemoryLines` 产出的字符串包含这些章节(`src/memdir/memdir.ts:199-266`):

```
# auto memory

You have a persistent, file-based memory system at `<dir>`. <DIR_EXISTS_GUIDANCE>

You should build up this memory system over time …

## Types of memory                ← user / feedback / project / reference 四类
…
## What NOT to save in memory     ← 反例清单(代码模式、git 历史、ephemeral 状态…)
…
## How to save memories           ← Step 1 写 frontmatter,Step 2 在 MEMORY.md 加索引
…
## When to access memories        ← 何时调取
…
## Before recommending from memory ← 推荐前的有效性校验
…
## Memory and other forms of persistence
                                   ← 跟 plan / task 的边界
…
## Searching past context …
```

**实际例子**:你在用的这个会话,系统提示词里就有这段(系统消息开头那部分 `# auto memory` 一直到 `## Memory and other forms of persistence`)——这就是 `memoryMechanicsPrompt` 渲染后的样子。

### 为什么"说明书"要进 system_prompt 而不是 user content

两个原因,都和 **prompt cache** 有关:

| 内容 | 变化频率 | 放进哪 | 命中 cache? |
|---|---|---|---|
| 机制说明书 | **几乎不变**(版本升级才动) | system_prompt | ✅ 命中 |
| 记忆条目 | 经常变(每天可能加几条) | user context attachment | ❌ 每次变就失效一小段 |

如果把记忆**内容**也塞进 system_prompt,记忆每加一条 cache 全失效,所有后续轮次都重新计费。所以 Claude Code 故意拆开:**机制说明走静态通道吃 cache,内容走动态通道在 turn 内注入**。

### 触发条件:不是每次都加

`QueryEngine.ts:316-319`:

```ts
const memoryMechanicsPrompt =
  customPrompt !== undefined && hasAutoMemPathOverride()
    ? await loadMemoryPrompt()
    : null
```

两个条件**同时满足**才会注入:
1. `customPrompt !== undefined` — SDK 调用者提供了**自定义 system prompt**(完整替换默认)
2. `hasAutoMemPathOverride()` — SDK 调用者还设置了 `CLAUDE_COWORK_MEMORY_PATH_OVERRIDE` 环境变量

**为什么是这两个条件**:
- 默认的 system prompt 里**本来就内嵌了** memory mechanics 的说明
- 但 SDK 调用者用了 `customSystemPrompt` 后,默认 prompt 整块被替换 → 机制说明丢了
- 此时如果调用者又显式开了 memory 目录(说明 ta 想用记忆),就把机制说明补回来——否则模型不知道怎么读写

也就是说:**这是为"自定义 prompt + 显式启用记忆"场景准备的补救逻辑**。普通 CLI 用户不走这条分支,默认 prompt 里已经有了。

### 还有几个相关分支

`loadMemoryPrompt` 内部按场景分发(`src/memdir/memdir.ts:419-490`):

| 场景 | 触发 | 产出 |
|---|---|---|
| **KAIROS 模式** | `feature('KAIROS') && getKairosActive()` | `buildAssistantDailyLogPrompt` — 每日日志型记忆 |
| **Team memory** | `feature('TEAMMEM') && isTeamMemoryEnabled()` | `buildCombinedMemoryPrompt` — auto + team 双目录 |
| **普通 auto memory** | `isAutoMemoryEnabled()` | `buildMemoryLines` 单目录 |
| **禁用** | 上面都不满足 | 返回 `null` |

外部还可以通过环境变量 `CLAUDE_COWORK_MEMORY_EXTRA_GUIDELINES` 追加策略文本到 mechanics 末尾(Anthropic Cowork 用的)。

## 对我们后端的启示

| Claude Code 做法 | 后端可借鉴 |
|---|---|
| **能力的"说明书"放静态通道,内容放动态通道** | 任何"模型能用某能力"的 prompt 里,把"怎么用"和"具体素材"拆成两段。说明书走 system prompt 吃 cache,素材走 user content 按需注入。**这是 prompt cache 友好设计的核心模式**。 |
| **同一个数据(记忆)用两个 API 暴露给模型**(`buildMemoryPrompt` 含内容 / `buildMemoryLines` 不含内容) | 对外的 prompt 渲染器**显式区分"含内容"和"只含说明"两种产出**,调用方明确知道自己拿到的是哪种,不会误用 |
| **机制说明跟 default prompt 解耦,自定义 prompt 时按需补救** | 我们的"自定义 system prompt"接口要文档化"哪些能力依赖默认 prompt 的某些段落",并提供 opt-in 把它们补回来。否则用户一开自定义 prompt,记忆/工具说明全废,体验是悬崖式跌落 |
| **`CLAUDE_COWORK_MEMORY_EXTRA_GUIDELINES` 让运营层追加策略** | 给"机制说明书"留一个外部追加点,运营/管理员可以叠加租户专属策略(例如"本租户禁止保存包含手机号的记忆"),**不用改代码也不用改默认 prompt** |
