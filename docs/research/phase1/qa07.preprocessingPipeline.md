# Q7. `query()` 里的预处理流水线具体是怎么走的?

> 配套阅读:[../phase1-agent-loop.md](../phase1-agent-loop.md) 第 4.2 节 · [qa06.submitMessageIsNotPromptBuilder](./qa06.submitMessageIsNotPromptBuilder.md) · [索引](./question.md)
>
> **提问背景**:Phase 1 笔记把这部分画成了「预处理流水线 (applyToolResultBudget → snip → microcompact → contextCollapse → autocompact)」,但没展开每层在做什么、为什么是这个顺序。

> ⚠️ 说明:这部分细节是 Phase 3(上下文与记忆管理)的核心主题。这里**只讲流水线骨架**——每层的位置、触发条件、产出形态。每层的内部算法(摘要怎么写、collapse 怎么选段落、cache_edits 怎么发到 API)留到 Phase 3。

## 预处理流水线的总体形态

每个 turn 进入 `query.ts:307` 的 `while(true)` 后,前 ~180 行(`query.ts:365-543`)就是预处理流水线。它跑在**每次** `callModel` 之前——也就是说,同一个 turn 内的第 1 次和第 3 次 callModel,都各自走过一遍这条流水线。

```
本轮 messagesForQuery = [...getMessagesAfterCompactBoundary(messages)]
              ↓
┌───── ① applyToolResultBudget ─────────────────────────────────────┐
│  超量的 tool_result 内容换成文件引用(spill-to-disk)              │
│  入口:utils/toolResultStorage.ts:924                              │
│  触发:contentReplacementState !== undefined(feature on)          │
│  作用域:cumulative—一旦换了 disk,后续 turn 一直看引用            │
│  产出:messagesForQuery 中部分 tool_result.content 变成短引用      │
└────────────────────────────────────────────────────────────────────┘
              ↓
┌───── ② snip(snipCompactIfNeeded)──────────────────────────────────┐
│  移除"僵尸"消息(用户撤销的 tool_use、孤立的 progress…)            │
│  入口:services/compact/snipCompact.ts(feature('HISTORY_SNIP'))   │
│  触发:HISTORY_SNIP 启用 + 满足启发式(比如 zombie tool_use 数量)  │
│  产出:剪后的 messagesForQuery + 一条 system snip_boundary 消息     │
│        + snipTokensFreed(供下游 autocompact 对账)                  │
└────────────────────────────────────────────────────────────────────┘
              ↓
┌───── ③ microcompact ──────────────────────────────────────────────┐
│  删除最老的 N 个 tool_result(只针对"可压缩工具":Read/Bash/Grep/  │
│  Glob/Web*/Edit/Write),换成占位串                                 │
│  入口:services/compact/microCompact.ts:253 (microcompactMessages) │
│  两种触发分路径:                                                  │
│    a. 时间触发(maybeTimeBasedMicrocompact):距上次 assistant 消息  │
│       超过阈值 → server cache 已凉,反正要重写,先精简内容          │
│    b. 缓存触发(cachedMicrocompactPath,ant-only):用 API 的 cache  │
│       editing 能力删旧 tool_result,本地 messages 不动              │
│  作用域:主线程(repl_main_thread*)+ 主模型 — fork agent 跳过      │
│  产出:同样 shape 的 messagesForQuery(可能附 pendingCacheEdits)   │
└────────────────────────────────────────────────────────────────────┘
              ↓
┌───── ④ contextCollapse(feature('CONTEXT_COLLAPSE'))──────────────┐
│  段落级折叠:把某些"读完不需要再看"的历史段(Glob 结果、Read 全文)│
│  替换成摘要,但**只折叠投影,REPL 仍能看完整版**                    │
│  入口:services/contextCollapse/index.ts:applyCollapsesIfNeeded    │
│  产出:projected messagesForQuery(折叠后)+ collapse 提交记录      │
│  与 autocompact 互斥:collapse 启用时 autocompact 自动让位         │
└────────────────────────────────────────────────────────────────────┘
              ↓
┌───── ⑤ autocompact(autoCompactIfNeeded)──────────────────────────┐
│  整段摘要 + 重建上下文,产出 compact_boundary 消息                  │
│  入口:services/compact/autoCompact.ts:241                         │
│  触发(shouldAutoCompact, autoCompact.ts:160-239):                │
│    • !DISABLE_COMPACT && autoCompactEnabled 设置                  │
│    • querySource 不是 session_memory / compact / marble_origami    │
│    • REACTIVE_COMPACT 没接管 / CONTEXT_COLLAPSE 没接管             │
│    • tokenCountWithEstimation(messages) - snipTokensFreed ≥ 阈值   │
│  代价:**自己要调一次 LLM** 写摘要(成本最高)                       │
│  产出:postCompactMessages → 整段替换 messagesForQuery               │
│         + 一条 system compact_boundary 消息(SDK 透传)              │
└────────────────────────────────────────────────────────────────────┘
              ↓
┌───── ⑥ blocking_limit 闸门 ───────────────────────────────────────┐
│  上面所有压缩都跑完了还超阈值 → yield API 错误消息 + return         │
│  Terminal('blocking_limit')                                       │
│  跳过条件:本轮已 autocompact / 是 compact/session_memory 子查询 /  │
│            reactiveCompact 接管 / contextCollapse 接管              │
└────────────────────────────────────────────────────────────────────┘
              ↓
       构造 fullSystemPrompt = appendSystemContext(systemPrompt, systemContext)
              ↓
            deps.callModel({ systemPrompt, messages: messagesForQuery, ... })
```

## 为什么是这个顺序

**代价从低到高,先用便宜的能"够"就别动贵的**。

| 层 | 代价 | 信息损失 | 何时一定要跑 |
|---|---|---|---|
| ① applyToolResultBudget | 极低(磁盘 I/O) | **可逆**(spill 出去,引用还在,需要再读回) | feature on 就跑 |
| ② snip | 低(纯客户端剪切) | 中(被剪的消息回不来,但本来就是僵尸) | 启发式触发 |
| ③ microcompact | 低 / 零(cached MC 走 API 缓存编辑,**不动 local messages**) | 中(老 tool_result 看不到了,但工具调用本身还在) | 时间触发 / 数量触发 |
| ④ contextCollapse | 低 - 中(折叠规则纯客户端) | 中 - 高(折叠掉的段以摘要呈现;但 REPL 仍能展开) | 配置开启 |
| ⑤ autocompact | **高(要单独调一次 LLM 写摘要)** | 高(整段历史变摘要,粒度全丢) | token 超阈值兜底 |

设计意图:**autocompact 是最后兜底**,如果前面四层把 token 数砍下来,根本不会走到第五层。这跟"先 GC 软引用、再 GC 弱引用、最后 stop-the-world"是同一种思路。

## 几个关键约束

### 1. 流水线**只为 callModel 服务**,不改 `state.messages`

`query.ts:365`:
```ts
let messagesForQuery = [...getMessagesAfterCompactBoundary(messages)]
```

`messagesForQuery` 是个**临时数组**,这条流水线产出的所有修改只在它身上。**`state.messages`(也即下一轮 turn 看到的那个数组)在这里没改**——它是在阶段 4(工具执行完)按 `[...messagesForQuery, ...assistantMessages, ...toolResults]` 重组的。

但 autocompact 例外——它产出的 `postCompactMessages` 会**整体替换** `messagesForQuery`,而且**产出 `compact_boundary` 消息会被推回上层** state.messages,后续 `getMessagesAfterCompactBoundary(...)` 就会过滤掉 boundary 之前的所有消息。这就是 compaction "持久化"的实现机制。

### 2. 互斥关系(`shouldAutoCompact` 的几个 early return)

`autoCompact.ts:160-239` 里这几个分支决定了**多层不会同时跑**:

| 互斥分支 | 条件 | 含义 |
|---|---|---|
| `querySource === 'session_memory' \|\| 'compact'` | fork agent 自己 | 防止"compact agent 自己又触发 autocompact"死循环 |
| `querySource === 'marble_origami'` | 上下文-agent | 它写到 module-level state,autocompact 触发 cleanup 会把主线程的 collapse log 也炸了 |
| `REACTIVE_COMPACT && tengu_cobalt_raccoon` | reactive 接管 | 改让"先打 API,413 回来再 compact"——比预判更准 |
| `CONTEXT_COLLAPSE && isContextCollapseEnabled()` | collapse 接管 | collapse 是"自己一套头寸管理",autocompact 会跟它打架 |

**设计教训**:不要"所有压缩同时上"。**显式互斥,谁先谁后,谁让谁,代码上要标注清楚**。

### 3. `snipTokensFreed` 给下游对账

`shouldAutoCompact` 第 167 行:
```ts
async function shouldAutoCompact(..., snipTokensFreed = 0) {
  ...
  const tokenCount = tokenCountWithEstimation(messages) - snipTokensFreed
}
```

为什么要传?——snip 删了一批消息,但 `tokenCountWithEstimation` 是从**剩余消息里最末的 assistant message 的 usage 字段**估算的(那个消息里报的是 snip 之前的窗口大小)。**所以本地估算永远滞后于实际**。snip 自己最清楚自己剪了多少 token,显式传给 autocompact 才不会"明明已经够小,还触发 autocompact 浪费一次 LLM"。

**设计教训**:**算上下文 token 数是个滞后量**。每一层压缩都应该自报"我省了多少",别让下游瞎猜。

### 4. blocking_limit 闸门的"反例"逻辑

`query.ts:628-648` 的 `if` 条件挺反直觉:

```ts
if (
  !compactionResult &&                       // 本轮没 autocompact
  querySource !== 'compact' &&
  querySource !== 'session_memory' &&
  !(reactiveCompact?.isReactiveCompactEnabled() && isAutoCompactEnabled()) &&
  !collapseOwnsIt
) {
  const { isAtBlockingLimit } = calculateTokenWarningState(...)
  if (isAtBlockingLimit) { yield 错误 → return }
}
```

"**这些情况都不闸门**"——为什么?

- 本轮已 autocompact:刚精简完,再闸门没意义
- compact/session_memory 子查询:它们就是用来"减小上下文"的,闸门它们就死锁了
- reactiveCompact 接管:让 API 真的 413 回来再处理,先放行
- collapseOwnsIt:同上,让 collapse 自己管

**设计教训**:**给"清理路径"开后门**。GC 时不要 GC GC 自己。

## 对我们后端的启示

| Claude Code 做法 | 后端可借鉴 |
|---|---|
| **多层压缩按代价从低到高排队,前面够用就别动后面** | 我们做长上下文管理时,**不要只做一个"超过阈值就 summarize"**。先做便宜的(spill 长结果到对象存储,留引用)、再做剪除(撤销的 tool call、孤立的 progress)、最后才上"用 LLM 写摘要"。每一层独立配置/独立 feature flag |
| **每次 callModel 前都走流水线,而不是入口处理一次** | **上下文压力是动态量**——tool 跑完返回 1MB 的 grep 结果,瞬间从"健康"跌到"濒危"。流水线必须在每次 API 调用前都重新跑一遍 |
| **`messagesForQuery` 临时数组,跟 `state.messages` 分开** | 别在原始消息数组上原地改,**为每次 callModel 构造一份临时投影**。出错回滚干净,boundary 消息只在确认成功时才推回 state.messages |
| **`snipTokensFreed` 模式:每层压缩自报省了多少** | 任何压缩层都该返回 `{ messages, tokensFreed }`,下游用这个对账,不要靠下游从消息里反推。**滞后估算是 bug 的温床** |
| **互斥关系(reactiveCompact/contextCollapse 接管时 autocompact 让位)显式编码,带注释解释 why** | 多种压缩策略并存时,**互斥不要靠默契**,代码里 `if (xxxOwnsIt) return false` 显式短路,注释解释"为什么这两个不能同时上"——后人修改时不会瞎拆 |
| **"清理路径"自身要绕过闸门** | 任何"用来减小负载"的路径(compact、cleanup、GC),自身要标记后绕过"过载保护",否则系统过载时反而进不去清理路径,**死锁** |
| **`QueryDeps` 注入 callModel/microcompact/autocompact** | 后端的 turn loop 接收一个 `deps` 对象注入这几个"昂贵且容易变化"的能力,**测试时换 fake、生产时换 fallback、AB 实验时换不同实现**,都不用改主流程代码 |
