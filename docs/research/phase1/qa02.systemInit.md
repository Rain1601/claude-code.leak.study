# Q2. `system_init` 消息是 system_prompt 和 processUserInput 加工后的最终 prompt 吗?

> 配套阅读:[../phase1-agent-loop.md](../phase1-agent-loop.md) · [索引](./question.md)
>
> **提问背景**:笔记第 0 节流程图里写了「发出 system_init 消息」,直觉容易把它当成"喂给模型的 system prompt"。

## 回答

**不是。两者完全是两条独立路径,只是名字撞了"system"这个词。**

| | `system_init` 消息 | system_prompt(喂给 LLM 的) |
|---|---|---|
| **是什么** | SDK 协议的一种 `SDKMessage`,subtype 为 `'init'` | 真正发给 Anthropic API 的 `system` 字段文本 |
| **去向** | yield 出去给 **SDK 消费者**(远程客户端、REPL bridge、SDK caller) | 走 `deps.callModel({ systemPrompt: fullSystemPrompt, ... })` 发给 **LLM** |
| **内容** | **元数据清单**(工具名列表、模型名、cwd、session_id、可用 slash 命令名、MCP server 状态、CLI 版本…) | **prompt 文本**(default system prompt + memory + appendSystemPrompt 拼好的字符串数组) |
| **频率** | 每 turn 开始时 yield 一次 | 每次 API 调用时随请求发 |
| **谁看** | 客户端 UI 用来渲染选择器、显示模型标识、gate 功能 | 模型 |

### `system_init` 长什么样

`buildSystemInitMessage`(`src/utils/messages/systemInit.ts:53-96`)产出的对象:

```ts
{
  type: 'system',
  subtype: 'init',
  cwd,                        // 当前工作目录
  session_id,
  tools: [...工具名],          // sdkCompatToolName 翻译过(Agent→Task 兼容)
  mcp_servers: [{ name, status }, ...],
  model,                      // 模型名(例如 'claude-opus-4-7')
  permissionMode,             // 'default' | 'plan' | 'acceptEdits' | 'bypassPermissions'
  slash_commands: [...用户可调用的命令名],
  apiKeySource,               // 'env' | 'config' | 'keychain' …
  betas,                      // 服务端 beta 标签
  claude_code_version,
  output_style,
  agents: [...agent type],
  skills: [...skill 名],
  plugins: [{ name, path, source }, ...],
  fast_mode_state,
  uuid,
}
```

注意里面**没有任何 prompt 文本**——只是清单。注释原话(`systemInit.ts:42-44`):

> The first message on the SDK stream carrying **session metadata** (cwd, tools, model, commands, etc.) that remote clients use to **render pickers and gate UI**.

### 真正的 system_prompt 在哪里发

回到 `QueryEngine.ts:321-325`,这里是 system_prompt 的拼装:

```ts
const systemPrompt = asSystemPrompt([
  ...(customPrompt !== undefined ? [customPrompt] : defaultSystemPrompt),
  ...(memoryMechanicsPrompt ? [memoryMechanicsPrompt] : []),
  ...(appendSystemPrompt ? [appendSystemPrompt] : []),
])
```

然后这个 `systemPrompt` 字符串数组通过 `query()` 的 `params.systemPrompt` 一路传给 `callModel`(`query.ts:449-451` 又拼了一层 `appendSystemContext`),最终在 API request body 的 `system` 字段里。**它从来不会以 `SDKMessage` 形式 yield 给 SDK 消费者**。

### 谁会消费 `system_init`

仓库里有 4 个消费者(`grep "subtype === 'init'"`):
- `src/hooks/useSSHSession.ts:79` — SSH 远程会话拿来初始化 UI 状态
- `src/hooks/useDirectConnect.ts:73` — 直连会话同上
- `src/hooks/useRemoteSession.ts:195` — 远程会话同上
- `src/remote/sdkMessageAdapter.ts:229` — Remote 协议适配器

它们干的事都是:**收到 init 后,把里面的工具列表、命令列表、模型等灌到客户端状态里,渲染下拉选择器和功能开关**。

### processUserInput 加工后的"最终 prompt"在哪里

不存在所谓"最终 prompt"这个单一对象。**processUserInput 加工的是 user message,不是 system message**。它产出的是 `messages: Message[]`(可能包含 UserMessage + AttachmentMessage[]),这些消息会:
1. push 到 `this.mutableMessages`(`QueryEngine.ts:431`)
2. 在每次 turn 迭代时被 `getMessagesAfterCompactBoundary()` 截取(`query.ts:365`),传给 `callModel({ messages, systemPrompt, ... })`

API 看到的最终请求 = `system: systemPrompt` + `messages: 历次 turn 累积的 messages`。这是两条独立的轴。

### 一句话区分

- **`system_init`** = SDK 协议握手包,**给客户端 UI 用**,告诉它"这次 turn 有哪些工具/命令/agent 可用、模型是谁、cwd 在哪"
- **`system_prompt`** = 喂给 LLM 的指令文本,**给模型看**,告诉它"你是 Claude Code,这是你的工具说明、这是用户的记忆、这是用户追加的策略"

两者唯一的共同点是开头都带 "system" 这个词。

## 对照源码

- `src/utils/messages/systemInit.ts:29-96` — `SystemInitInputs` 类型与 `buildSystemInitMessage` 实现
- `src/QueryEngine.ts:540-551` — `yield buildSystemInitMessage(...)` 调用点
- `src/QueryEngine.ts:288-325` — 真正的 system_prompt 在这里拼装(`fetchSystemPromptParts` + `asSystemPrompt`)
- `src/query.ts:449-451` — `fullSystemPrompt = appendSystemContext(systemPrompt, systemContext)`,再叠加一层动态上下文(cwd、git 状态等)
- `src/query.ts:659-708` — `deps.callModel({ systemPrompt: fullSystemPrompt, messages: ..., ... })` 真正发请求
- `src/hooks/useSSHSession.ts:79` 等 — 客户端消费 `subtype === 'init'`

## 对我们后端的启示

| Claude Code 做法 | 后端可借鉴 |
|---|---|
| **每 turn 开头给客户端发一条 init 元数据消息** | 我们的 SSE/WebSocket 协议第一帧也应当是元数据帧(本次请求实际用了哪些工具、哪个模型、session_id 是什么),**前端就不用从其他渠道反查这些上下文** |
| **`sdkCompatToolName` 做向后兼容映射** | 工具改名时,**对外协议先维持旧名一段时间**(comment 里写了 TODO(next-minor)),客户端版本升级缓冲期友好 |
| **`userInvocable !== false` 才出现在 init 里** | 内部用的命令不暴露给前端选择器。**对外协议是受控视图**,不是数据库 dump |
| **API 元数据(model、cwd、tools)和 LLM 指令(system_prompt)分两条通道** | 后端**不要**把"模型看到的 system prompt"原文回流给前端——既泄露策略又制造混淆。前端要"模型是谁"就明确给个 `model` 字段,而不是从 prompt 文本里 parse |
